# Estado del proyecto CAM e imputacion general

## Objetivo del documento

Este documento resume el estado actual del proyecto ETL CAM y explica el siguiente paso: la imputacion general de variables para analisis de correlacion y modelos predictivos.

La idea principal es mejorar la interpretacion de los datos antes de construir mapas de correlacion o entrenar modelos. Para esto se necesita una base mas completa, pero sin perder la diferencia entre datos reales e imputados.

## Estado actual del proyecto

Hasta este punto el proyecto tiene las siguientes etapas construidas:

```text
1. Extraccion
2. Transformacion
3. Analisis de calidad temporal
4. Imputacion especifica de nivel_m
5. Interpretacion de resultados de imputacion de nivel_m
6. Estado del proyecto e imputacion general
7. EDA de series de tiempo
```

## Criterio sobre las salidas de imputacion

En el proyecto existen dos procesos relacionados con imputacion:

```text
imputacion_nivel.py
imputacion_general.py
```

No se deben interpretar como dos salidas finales que compiten entre si. Cumplen funciones distintas.

La imputacion especifica de `nivel_m` se conserva como una etapa metodologica y de validacion, porque `nivel_m` es la variable principal para el estudio de inundaciones. Sirve para entender con detalle que tan confiable queda la serie de nivel del agua.

La imputacion general es la salida integrada para analisis posteriores. Esta es la base que se debe usar para mapas de correlacion y modelos porque contiene `nivel_m` junto con las demas variables predictoras bajo una misma estructura.

Por tanto, la salida oficial para correlacion y modelado es:

```text
ETL_CAM/transform/data/imputacion_general/variables_modelo_imputadas.csv
```

La salida de apoyo metodologico es:

```text
ETL_CAM/transform/data/imputacion/nivel_m_imputado.csv
```

## Por que es necesaria la imputacion general

Para construir mapas de correlacion y modelos, se necesita que las variables tengan suficientes datos disponibles. Si una variable tiene muchos valores vacios, las correlaciones pueden calcularse con pocas filas y dar una interpretacion inestable.

Sin embargo, no se debe imputar todo sin control. Un valor imputado no tiene la misma confianza que un valor observado.

Por eso la imputacion general conserva informacion de calidad mediante columnas como:

```text
temperatura_c_fue_imputado
temperatura_c_calidad
lluvia_mm_fue_imputado
lluvia_mm_calidad
```

## Variables incluidas en la imputacion general

El script trabaja con:

```text
nivel_m
temperatura_c
humedad_pct
radiacion_solar_w_m2
lluvia_mm
direccion_viento_grados
velocidad_viento_m_s

```

Aunque `nivel_m` ya fue trabajado en una imputacion especifica, se incluye aqui para que la tabla final tenga todas las variables en el mismo archivo y con el mismo formato de calidad. Esto evita tener que cruzar manualmente varios archivos al momento de hacer correlaciones o entrenar modelos.

## Disponibilidad observada antes de imputar

La revision del archivo transformado mostro que la disponibilidad cambia mucho por estacion y variable.

Por ejemplo:

```text
La_Lindosa tiene 98,49% de nivel_m, pero no tiene humedad, radiacion, viento ni presion.
Los_Micos tiene 96,18% de nivel_m y 99,67% de temperatura.
El_Meson no tiene nivel_m, pero tiene buena disponibilidad de temperatura, humedad, lluvia, viento y presion.
Santa_Rosalia no tiene nivel_m, pero tiene buena disponibilidad meteorologica.
Puente_Bruselas tiene nivel_m y velocidad_agua_m_s, pero poca cobertura general.
```

Esto confirma que no todas las estaciones cumplen el mismo papel. Algunas son buenas como objetivo hidrologico y otras pueden servir como apoyo meteorologico.

## Reglas de imputacion usadas

### Variables numericas continuas

Para variables como:

```text
temperatura_c
humedad_pct
radiacion_solar_w_m2
velocidad_viento_m_s
presion_atmosferica_hpa
voltaje_bateria_vdc
velocidad_agua_m_s
```

se usa:

```text
1. Valor observado
2. Interpolacion temporal para huecos de hasta 60 minutos
3. Promedio historico por mes, dia, hora y minuto
4. Promedio historico por mes, hora y minuto
```

### Lluvia

Para `lluvia_mm` no se aplica interpolacion lineal directa.

La razon es que lluvia igual a cero y lluvia faltante no significan lo mismo. Un cero indica que no llovio; un vacio indica que no hubo medicion.

Por eso se usan promedios historicos, pero conservando la marca de calidad.

### Direccion del viento

Para `direccion_viento_grados` no se usa una interpolacion numerica normal. Esta variable es circular:

```text
359 grados y 1 grado estan cerca, no lejos.
```

Por eso el script trabaja con interpolacion circular y promedio circular.

## Calidad de imputacion

La calidad se interpreta asi:

```text
real: dato observado.
alta: dato interpolado en hueco corto.
media: dato estimado con promedio historico calendario.
baja: dato estimado con promedio historico mensual-horario.
sin_dato: no fue posible imputar.
```

## Como usar estos resultados para correlacion

Se recomienda generar dos mapas de correlacion:

```text
1. Correlacion con datos observados.
2. Correlacion con datos imputados.
```

Si ambos mapas se parecen, la imputacion no esta cambiando demasiado la relacion entre variables.

Si son muy diferentes, significa que la imputacion esta influyendo fuertemente y se debe interpretar con cuidado.

Para estos mapas, la base principal debe ser:

```text
variables_modelo_imputadas.csv
```

## Como usar estos resultados para modelos

Para modelos predictivos se recomienda:

- Usar `nivel_m` desde `variables_modelo_imputadas.csv` como variable objetivo solo en estaciones que si tienen datos reales de nivel.
- Incluir variables meteorologicas de estaciones auxiliares solo si tienen buena cobertura.
- No mezclar valores reales e imputados sin considerar la columna de calidad.
- Probar modelos con y sin variables de baja calidad.
