# Estado del proyecto CAM e imputación general

## Objetivo del documento

Este documento resume el estado actual del proyecto ETL CAM y explica el siguiente paso: la imputación general de variables para análisis de correlación y modelos predictivos.

La idea principal es mejorar la interpretación de los datos antes de construir mapas de correlación o entrenar modelos. Para esto se necesita una base más completa, pero sin perder la diferencia entre datos reales e imputados.

## Estado actual del proyecto

Hasta este punto el proyecto tiene las siguientes etapas construidas:

```text
1. Extracción
2. Transformación
3. Análisis de calidad temporal
4. Imputación específica de nivel_m
5. Interpretación de resultados de imputación de nivel_m
6. Estado del proyecto e imputación general
```

## Criterio sobre las salidas de imputación

En el proyecto existen dos procesos relacionados con imputación:

```text
imputacion_nivel.py
imputacion_general.py
```

No se deben interpretar como dos salidas finales que compiten entre sí. Cumplen funciones distintas.

La imputación específica de `nivel_m` se conserva como una etapa metodológica y de validación, porque `nivel_m` es la variable principal para el estudio de inundaciones. Sirve para entender con detalle qué tan confiable queda la serie de nivel del agua.

La imputación general es la salida integrada para análisis posteriores. Esta es la base que se debe usar para mapas de correlación y modelos porque contiene `nivel_m` junto con las demás variables predictoras bajo una misma estructura.

Por tanto, la salida oficial para correlación y modelado es:

```text
ETL_CAM/transform/data/imputacion_general/variables_modelo_imputadas.csv
```

La salida de apoyo metodológico es:

```text
ETL_CAM/transform/data/imputacion/nivel_m_imputado.csv
```

## Por qué es necesaria la imputación general

Para construir mapas de correlación y modelos, se necesita que las variables tengan suficientes datos disponibles. Si una variable tiene muchos valores vacíos, las correlaciones pueden calcularse con pocas filas y dar una interpretación inestable.

Sin embargo, no se debe imputar todo sin control. Un valor imputado no tiene la misma confianza que un valor observado.

Por eso la imputación general conserva información de calidad mediante columnas como:

```text
temperatura_c_fue_imputado
temperatura_c_calidad
lluvia_mm_fue_imputado
lluvia_mm_calidad
```

## Variables incluidas en la imputación general

El script trabaja con:

```text
nivel_m
temperatura_c
humedad_pct
radiacion_solar_w_m2
lluvia_mm
direccion_viento_grados
velocidad_viento_m_s

```

Aunque `nivel_m` ya fue trabajado en una imputación específica, se incluye aquí para que la tabla final tenga todas las variables en el mismo archivo y con el mismo formato de calidad. Esto evita tener que cruzar manualmente varios archivos al momento de hacer correlaciones o entrenar modelos.

## Disponibilidad observada antes de imputar

La revisión del archivo transformado mostró que la disponibilidad cambia mucho por estación y variable.

Por ejemplo:

```text
La_Lindosa tiene 98,49% de nivel_m, pero no tiene humedad, radiación, viento ni presión.
Los_Micos tiene 96,18% de nivel_m y 99,67% de temperatura.
El_Meson no tiene nivel_m, pero tiene buena disponibilidad de temperatura, humedad, lluvia, viento y presión.
Santa_Rosalia no tiene nivel_m, pero tiene buena disponibilidad meteorológica.
Puente_Bruselas tiene nivel_m y velocidad_agua_m_s, pero poca cobertura general.
```

Esto confirma que no todas las estaciones cumplen el mismo papel. Algunas son buenas como objetivo hidrológico y otras pueden servir como apoyo meteorológico.

## Reglas de imputación usadas

### Variables numéricas continuas

Para variables como:

```text
temperatura_c
humedad_pct
radiacion_solar_w_m2
velocidad_viento_m_s
presion_atmosferica_hpa
voltaje_bateria_vdc
velocidad_agua_m_s
```

se usa:

```text
1. Valor observado
2. Interpolación temporal para huecos de hasta 60 minutos
3. Promedio histórico por mes, día, hora y minuto
4. Promedio histórico por mes, hora y minuto
```

### Lluvia

Para `lluvia_mm` no se aplica interpolación lineal directa.

La razón es que lluvia igual a cero y lluvia faltante no significan lo mismo. Un cero indica que no llovió; un vacío indica que no hubo medición.

Por eso se usan promedios históricos, pero conservando la marca de calidad.

### Dirección del viento

Para `direccion_viento_grados` no se usa una interpolación numérica normal. Esta variable es circular:

```text
359 grados y 1 grado están cerca, no lejos.
```

Por eso el script trabaja con interpolación circular y promedio circular.

## Calidad de imputación

La calidad se interpreta así:

```text
real: dato observado.
alta: dato interpolado en hueco corto.
media: dato estimado con promedio histórico calendario.
baja: dato estimado con promedio histórico mensual-horario.
sin_dato: no fue posible imputar.
```

## Cómo usar estos resultados para correlación

Se recomienda generar dos mapas de correlación:

```text
1. Correlación con datos observados.
2. Correlación con datos imputados.
```

Si ambos mapas se parecen, la imputación no está cambiando demasiado la relación entre variables.

Si son muy diferentes, significa que la imputación está influyendo fuertemente y se debe interpretar con cuidado.

Para estos mapas, la base principal debe ser:

```text
variables_modelo_imputadas.csv
```

## Cómo usar estos resultados para modelos

Para modelos predictivos se recomienda:

- Usar `nivel_m` desde `variables_modelo_imputadas.csv` como variable objetivo solo en estaciones que sí tienen datos reales de nivel.
- Incluir variables meteorológicas de estaciones auxiliares solo si tienen buena cobertura.
- No mezclar valores reales e imputados sin considerar la columna de calidad.
- Probar modelos con y sin variables de baja calidad.
