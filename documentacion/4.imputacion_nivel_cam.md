# Documento del proceso de imputación de nivel CAM

## Objetivo

El objetivo de esta etapa es generar una serie de tiempo continua para la variable `nivel_m`, que es la variable principal del proyecto para el análisis y predicción de posibles inundaciones.

La imputación se realiza después de la extracción, transformación y análisis de calidad temporal. Esto permite evitar que los datos se rellenen sin conocer antes la frecuencia, los huecos y la disponibilidad real de la variable.

Esta etapa se conserva como análisis específico y respaldo metodológico de la variable objetivo. La base final integrada para mapas de correlación y modelos se genera posteriormente con `imputacion_general.py`.

## Ubicación del código

El código se encuentra en:

```text
ETL_CAM/transform/imputacion_nivel.py
```

## Entrada del proceso

El archivo de entrada es:

```text
ETL_CAM/transform/data/cam_transformado.csv
```

El script usa principalmente estas columnas:

```text
fecha_hora
estacion
nivel_m
```

## Salidas del proceso

El proceso genera:

```text
ETL_CAM/transform/data/imputacion/nivel_m_imputado.csv
ETL_CAM/transform/data/imputacion/resumen_imputacion_nivel.csv
```

Estas salidas no son la tabla final de modelado. Sirven para revisar y explicar con detalle la calidad de la imputación de `nivel_m`.

También genera un log:

```text
ETL_CAM/transform/imputacion_nivel.log
```

## Por qué se hace en una etapa separada

La imputación no se hizo dentro del transformador porque `nivel_m` pertenece a una serie de tiempo hidrológica. En este tipo de datos, rellenar valores faltantes puede cambiar el comportamiento de la serie, especialmente durante eventos de subida rápida del nivel del agua.

Por eso primero se limpia la información, después se analiza la continuidad temporal y finalmente se aplica una imputación controlada.

## Variable principal

La variable imputada es:

```text
nivel_m
```

El archivo de salida conserva el valor original y crea un valor imputado:

```text
nivel_m_original
nivel_m_imputado
```

Esto permite diferenciar los datos medidos por la estación de los datos estimados.

## Frecuencia objetivo

El análisis de calidad temporal detectó una frecuencia dominante de:

```text
5 minutos
```

Por eso el imputador construye una grilla temporal de 5 minutos para cada estación.

## Redondeo temporal

Algunas estaciones no siempre registran exactamente en minutos como `00`, `05`, `10`, etc. Por ejemplo, pueden aparecer registros en `00:26`, `00:31` o `23:58`.

Para evitar que esos pequeños desfases creen huecos falsos, el script redondea `fecha_hora` a la grilla más cercana de 5 minutos antes de imputar.

## Métodos de imputación

El proceso usa una estrategia por niveles.

### 1. Valor observado

Si existe un dato real de `nivel_m`, se conserva sin modificar.

Método asignado:

```text
observado
```

Calidad:

```text
real
```

### 2. Interpolación temporal

Si falta un valor dentro de un hueco corto, se aplica interpolación lineal temporal.

El límite definido es:

```text
60 minutos
```

Esto significa que solo se interpolan huecos pequeños, donde hay valores cercanos antes y después del faltante.

Método asignado:

```text
interpolacion_temporal_60min
```

Calidad:

```text
alta
```

### 3. Promedio histórico calendario

Si después de la interpolación todavía falta un valor, el script intenta usar el promedio histórico de la misma estación para la misma combinación:

```text
mes
dia
hora
minuto
```

Este método se parece a la idea de comparar con años anteriores, porque busca cómo se comportó la misma estación en la misma fecha y hora de otros años.

Para aplicarlo se requieren al menos dos observaciones históricas.

Método asignado:

```text
promedio_historico_mes_dia_hora_minuto
```

Calidad:

```text
media
```

### 4. Promedio histórico mensual-horario

Si no hay suficientes datos para el promedio calendario exacto, se usa un promedio más general por:

```text
mes
hora
minuto
```

Este método es menos preciso porque no usa el día exacto, pero mantiene el patrón horario y estacional.

Método asignado:

```text
promedio_historico_mes_hora_minuto
```

Calidad:

```text
baja
```

### 5. Sin imputar

Si no hay información suficiente para estimar un valor, el dato permanece vacío.

Método asignado:

```text
sin_imputar
```

Calidad:

```text
sin_dato
```

## Columnas del archivo imputado

El archivo `nivel_m_imputado.csv` contiene:

```text
fecha_hora
estacion
nivel_m_original
nivel_m_imputado
fue_imputado_nivel
metodo_imputacion_nivel
calidad_imputacion_nivel
```

## Interpretación de las columnas

### nivel_m_original

Valor real registrado por la estación. Si está vacío, significa que no había medición original para ese instante.

### nivel_m_imputado

Valor final para usar en análisis posteriores. Puede ser un valor observado o un valor estimado.

### fue_imputado_nivel

Indica si el valor fue estimado por el proceso de imputación.

### metodo_imputacion_nivel

Indica cómo se obtuvo el valor final.

### calidad_imputacion_nivel

Clasifica la confianza del valor:

```text
real
alta
media
baja
sin_dato
```

## Estaciones sin nivel_m

Si una estación no tiene ningún dato disponible de `nivel_m`, el script la omite para la imputación de nivel. Esto queda registrado en el resumen.

Estas estaciones no deben usarse como variable objetivo para un modelo de inundaciones basado en nivel del agua, aunque pueden servir como estaciones auxiliares si tienen otras variables útiles.

## Comando de ejecución

Desde la raíz del proyecto:

```powershell
python ETL_CAM\transform\imputacion_nivel.py
```

## Recomendación para el modelo

Para modelos de predicción de inundaciones, la base oficial debe ser la generada por:

```text
ETL_CAM/transform/imputacion_general.py
```

La salida `nivel_m_imputado.csv` se usa como respaldo para interpretar la variable objetivo. Los valores con calidad `real` y `alta` son los más confiables. Los valores de calidad `media` o `baja` deben usarse con cuidado, especialmente si corresponden a períodos de crecida o eventos extremos.
