# Documento del proceso de imputacion de nivel CAM

## Objetivo

El objetivo de esta etapa es generar una serie de tiempo continua para la variable `nivel_m`, que es la variable principal del proyecto para el analisis y prediccion de posibles inundaciones.

La imputacion se realiza despues de la extraccion, transformacion y analisis de calidad temporal. Esto permite evitar que los datos se rellenen sin conocer antes la frecuencia, los huecos y la disponibilidad real de la variable.

Esta etapa se conserva como analisis especifico y respaldo metodologico de la variable objetivo. La base final integrada para mapas de correlacion y modelos se genera posteriormente con `imputacion_general.py`.

## Ubicacion del codigo

El codigo se encuentra en:

```text
ETL_CAM/transform/imputacion_nivel.py
```

## Entrada del proceso

El archivo de entrada es:

```text
ETL_CAM/transform/data/cam_transformado.csv
```

El script usa principalmente estas columnas:

```text
fecha_hora
estacion
nivel_m
```

## Salidas del proceso

El proceso genera:

```text
ETL_CAM/transform/data/imputacion/nivel_m_imputado.csv
ETL_CAM/transform/data/imputacion/resumen_imputacion_nivel.csv
```

Estas salidas no son la tabla final de modelado. Sirven para revisar y explicar con detalle la calidad de la imputacion de `nivel_m`.

Tambien genera un log:

```text
ETL_CAM/transform/imputacion_nivel.log
```

## Por que se hace en una etapa separada

La imputacion no se hizo dentro del transformador porque `nivel_m` pertenece a una serie de tiempo hidrologica. En este tipo de datos, rellenar valores faltantes puede cambiar el comportamiento de la serie, especialmente durante eventos de subida rapida del nivel del agua.

Por eso primero se limpia la informacion, despues se analiza la continuidad temporal y finalmente se aplica una imputacion controlada.

## Variable principal

La variable imputada es:

```text
nivel_m
```

El archivo de salida conserva el valor original y crea un valor imputado:

```text
nivel_m_original
nivel_m_imputado
```

Esto permite diferenciar los datos medidos por la estacion de los datos estimados.

## Frecuencia objetivo

El analisis de calidad temporal detecto una frecuencia dominante de:

```text
5 minutos
```

Por eso el imputador construye una grilla temporal de 5 minutos para cada estacion.

## Redondeo temporal

Algunas estaciones no siempre registran exactamente en minutos como `00`, `05`, `10`, etc. Por ejemplo, pueden aparecer registros en `00:26`, `00:31` o `23:58`.

Para evitar que esos pequenos desfases creen huecos falsos, el script redondea `fecha_hora` a la grilla mas cercana de 5 minutos antes de imputar.

## Metodos de imputacion

El proceso usa una estrategia por niveles.

### 1. Valor observado

Si existe un dato real de `nivel_m`, se conserva sin modificar.

Metodo asignado:

```text
observado
```

Calidad:

```text
real
```

### 2. Interpolacion temporal

Si falta un valor dentro de un hueco corto, se aplica interpolacion lineal temporal.

El limite definido es:

```text
60 minutos
```

Esto significa que solo se interpolan huecos pequenos, donde hay valores cercanos antes y despues del faltante.

Metodo asignado:

```text
interpolacion_temporal_60min
```

Calidad:

```text
alta
```

### 3. Promedio historico calendario

Si despues de la interpolacion todavia falta un valor, el script intenta usar el promedio historico de la misma estacion para la misma combinacion:

```text
mes
dia
hora
minuto
```

Este metodo se parece a la idea de comparar con anos anteriores, porque busca como se comporto la misma estacion en la misma fecha y hora de otros anos.

Para aplicarlo se requieren al menos dos observaciones historicas.

Metodo asignado:

```text
promedio_historico_mes_dia_hora_minuto
```

Calidad:

```text
media
```

### 4. Promedio historico mensual-horario

Si no hay suficientes datos para el promedio calendario exacto, se usa un promedio mas general por:

```text
mes
hora
minuto
```

Este metodo es menos preciso porque no usa el dia exacto, pero mantiene el patron horario y estacional.

Metodo asignado:

```text
promedio_historico_mes_hora_minuto
```

Calidad:

```text
baja
```

### 5. Sin imputar

Si no hay informacion suficiente para estimar un valor, el dato permanece vacio.

Metodo asignado:

```text
sin_imputar
```

Calidad:

```text
sin_dato
```

## Columnas del archivo imputado

El archivo `nivel_m_imputado.csv` contiene:

```text
fecha_hora
estacion
nivel_m_original
nivel_m_imputado
fue_imputado_nivel
metodo_imputacion_nivel
calidad_imputacion_nivel
```

## Interpretacion de las columnas

### nivel_m_original

Valor real registrado por la estacion. Si esta vacio, significa que no habia medicion original para ese instante.

### nivel_m_imputado

Valor final para usar en analisis posteriores. Puede ser un valor observado o un valor estimado.

### fue_imputado_nivel

Indica si el valor fue estimado por el proceso de imputacion.

### metodo_imputacion_nivel

Indica como se obtuvo el valor final.

### calidad_imputacion_nivel

Clasifica la confianza del valor:

```text
real
alta
media
baja
sin_dato
```

## Estaciones sin nivel_m

Si una estacion no tiene ningun dato disponible de `nivel_m`, el script la omite para la imputacion de nivel. Esto queda registrado en el resumen.

Estas estaciones no deben usarse como variable objetivo para un modelo de inundaciones basado en nivel del agua, aunque pueden servir como estaciones auxiliares si tienen otras variables utiles.

## Comando de ejecucion

Desde la raiz del proyecto:

```powershell
python ETL_CAM\transform\imputacion_nivel.py
```

## Recomendacion para el modelo

Para modelos de prediccion de inundaciones, la base oficial debe ser la generada por:

```text
ETL_CAM/transform/imputacion_general.py
```

La salida `nivel_m_imputado.csv` se usa como respaldo para interpretar la variable objetivo. Los valores con calidad `real` y `alta` son los mas confiables. Los valores de calidad `media` o `baja` deben usarse con cuidado, especialmente si corresponden a periodos de crecida o eventos extremos.
